Session 1

Graph Database Fundamentals

What is a Graph Database?

  • Definition: Database using graph structures for semantic queries
  • Core Philosophy: Relationships are first-class citizens
  • Storage Model: Native graph storage optimized for connections
  • Query Focus: Pattern-based queries instead of table joins
Key Insight: Traditional databases store relationships in separate tables. Graph databases store relationships as direct connections.

Graph Theory Foundations

  • Mathematical Roots: Based on graph theory from mathematics
  • Vertices & Edges: Nodes and relationships in database terms
  • Directed vs Undirected: Relationships can have direction
  • Weighted Graphs: Relationships can have properties
Real-World Mapping:
Mathematical Graph → Database Implementation
Vertex → Node | Edge → Relationship | Weight → Property

Core Graph Components: Nodes

  • Definition: Represent entities, objects, or concepts
  • Labels: Categorize nodes by type (Person, Product)
  • Properties: Key-value pairs storing attributes
  • Multiple Labels: Nodes can have multiple classifications
Examples:
Person {name: "Alice", age: 28}
Product {name: "iPhone", price: 999}
Location {name: "Central Park", area: 843}

Core Graph Components: Relationships

  • Definition: Connections with semantic meaning
  • Type: Every relationship has exactly one type
  • Direction: Have start and end nodes
  • Properties: Store metadata about connections
Examples:
(Alice)-[:KNOWS {since: 2020}]->(Bob)
(User)-[:PURCHASED {date: "2024-01-15"}]->(Product)

Properties: Adding Rich Context

  • Data Types: String, Integer, Float, Boolean, Date, Arrays
  • Schema-Optional: Properties added dynamically
  • Indexing: Properties can be indexed for fast lookups
  • Constraints: Uniqueness and existence constraints
Flexibility: Unlike SQL schemas, add new properties without altering structures.

Why Graph Databases? - Problem Statement

  • Relational Limitations: Complex joins become expensive
  • Relationship Depth: Multi-level relationships cumbersome
  • Schema Rigidity: Difficult to evolve data models
  • Performance Degradation: Slower with relationship depth
SQL Challenge: Finding friends-of-friends requires multiple self-joins, becoming complex and slow.

Performance Advantages

Relationship Traversal

O(1) vs O(log n) in relational

No Complex Joins

Direct pointer navigation

Variable Depth

Efficient recursive patterns

Real-time Analysis

Interactive exploration

Performance Comparison

Query Type SQL Approach Graph Approach Performance
Direct Link Simple JOIN Single hop Similar
Friends of Friends Self-JOIN 2-hop pattern 10x faster
3+ Degrees Multiple JOINs Variable path 100x+ faster

When to Choose Graph Databases

  • Relationship-Heavy Data: Connections as important as entities
  • Complex Queries: Multi-hop traversals and patterns
  • Real-time Requirements: Interactive analysis
  • Evolving Schema: Flexible models that change
  • Network Analysis: Graph algorithms and science
Decision Rule: If queries ask "who knows whom" or "what connects to what," consider graphs.

Neo4j Architecture & Development Environment

Neo4j: The Leading Graph Database

  • Market Leader: Most popular graph database worldwide
  • ACID Compliance: Full transactional support
  • Native Graph Storage: Purpose-built for graphs
  • Cypher Query Language: Declarative, SQL-inspired
  • Enterprise Ready: Clustering, security, monitoring
Why Neo4j: Combines NoSQL flexibility with database reliability, optimized for connected data.

Neo4j Architecture Components

Core Engine Stack

  • Graph Engine: Native processing and storage
  • Cypher Engine: Query planning and execution
  • Storage Layer: Optimized file structures
  • Transaction Layer: ACID properties and concurrency
  • Index Layer: Schema and full-text search

Neo4j Storage Architecture

  • Node Store: Fixed-size records for metadata
  • Relationship Store: Doubly-linked lists
  • Property Store: Key-value storage
  • String Store: Optimized string values
  • Index Files: B-tree and full-text indexes
Performance Benefit: Direct pointer navigation means O(1) traversal regardless of database size.

Enterprise Edition

  • Clustering and high availability
  • Advanced security (LDAP, SSO)

Desktop Benefits: Professional development environment with project management and plugin ecosystem.

Session 3

Cypher Query Language

Introduction to Cypher

  • Declarative Language: Describe what you want
  • ASCII Art Syntax: Visual graph representation
  • SQL-Inspired Keywords: MATCH, WHERE, RETURN
  • Pattern Matching: Express traversals intuitively
  • Human Readable: Reads like natural language
Core Philosophy:
(node)-[:RELATIONSHIP]->(node) visually represents connections

Cypher Syntax Fundamentals

  • Nodes: Represented by parentheses (node)
  • Relationships: Brackets with direction [:TYPE]->
  • Variables: Bind elements for reference
  • Labels: Categorize with :Label syntax
  • Properties: Key-value pairs in {key: value}
Pattern Examples:
(n) - any node | (p:Person) - person node
(a)-[:KNOWS]->(b) - knows relationship

CREATE: Building Graph Foundation

Basic Node Creation:
CREATE (p:Person {name: "Alice", age: 25})

Multiple Nodes:
CREATE (a:Person {name: "Alice"}), (b:Person {name: "Bob"})

Multiple Labels:
CREATE (e:Person:Employee {name: "Charlie"})

CREATE: Building Relationships

Simple Relationship:
CREATE (a:Person)-[:KNOWS]->(b:Person)

With Properties:
CREATE (a)-[:KNOWS {since: 2020}]->(b)

Complete Pattern:
CREATE (alice:Person {name: "Alice"})-[:KNOWS]->(bob:Person {name: "Bob"})

MATCH: Finding Graph Patterns

Find All Nodes:
MATCH (n) RETURN n

Find by Label:
MATCH (p:Person) RETURN p

Find Relationships:
MATCH (a:Person)-[:KNOWS]->(b:Person) RETURN a, b

WHERE: Powerful Filtering

Property Conditions:
MATCH (p:Person) WHERE p.age > 25 RETURN p

String Operations:
MATCH (p:Person) WHERE p.name STARTS WITH "A" RETURN p

Pattern Filters:
MATCH (p:Person) WHERE (p)-[:KNOWS]->() RETURN p

RETURN: Shaping Output

Return Properties:
MATCH (p:Person) RETURN p.name, p.age

With Aliases:
MATCH (p:Person) RETURN p.name AS name

Ordering and Limiting:
MATCH (p:Person) RETURN p ORDER BY p.age LIMIT 5

Labels and Relationship Types

Node Labels

  • Use CamelCase (Person, Employee)
  • Multiple labels for inheritance
  • Enable schema and indexing
  • Improve query performance

Relationship Types

  • Use UPPER_CASE with underscores
  • Use verbs (KNOWS, WORKS_FOR)
  • Be directionally meaningful
  • One type per relationship
